# LiteLLM: Project-Specific Analysis & SOP

## Architecture Summary

LiteLLM Proxy exposes multiple model providers behind an OpenAI-compatible API.
This harness treats LiteLLM as an external execution backend and focuses on
agentic code and workflow mutation loops.

Default deployment assumptions in this customization branch:
- Proxy URL: `https://litellm.nuriy.com/v1`
- Default model alias: `gpt-5.4`
- API key stays external via env or local config, not committed to repo

## CLI Strategy

This harness is not a generic chat CLI. The primary default job is:

1. Load a repo-local YAML task or flow.
2. Gather target file context from a workspace.
3. Ask a LiteLLM model alias for structured edit actions.
4. Generate unified diff + machine-readable action JSON.
5. Auto-apply repo-scoped edits.
6. Run verification commands, using task-configured commands first and heuristics otherwise.
7. Repeat until success criteria or iteration limits are reached.

## Command Map

| Intent | CLI |
|---|---|
| Run one agent task | `cli task run task.yaml --workspace .` |
| Validate task/flow YAML | `cli flow validate flow.yaml` |
| Replay a multi-step flow | `cli flow run flow.yaml --workspace .` |
| Check proxy status | `cli health` |
| List configured aliases | `cli models list` |
| Show last patch diff | `cli patch show` |
| Export last patch JSON/diff | `cli patch export outdir/` |
| Roll back last patch | `cli patch rollback` |
| Inspect run history | `cli session history` |

## YAML Shape

```yaml
name: repair-loop-controller
goal: Fix the orchestration loop to stop when retries are exhausted.
target:
  include:
    - src/engine.py
    - flows/retry.yaml
model: my-best-coder
success_criteria:
  - Retry loop stops on max attempts.
verify:
  command: pytest tests/test_engine.py -q
max_iterations: 3
patch_policy:
  auto_apply: true
steps:
  - prompt: Inspect the loop and patch only the target files.
```

## Safety Posture

- Writes are allowed only inside the declared workspace.
- Absolute paths outside the workspace are rejected.
- Destructive shell actions are not generated by the harness.
- Verification runs after each patch by default.
- The default optimization target is app editing, especially agent execution engines, flows, and loops.

## Tests

- Unit tests mock LiteLLM proxy responses and shell verification.
- E2E-style tests exercise analyze -> patch -> apply -> verify loops on temp repos.
